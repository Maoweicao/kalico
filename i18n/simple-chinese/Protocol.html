    协议¶

    Kalico 消息协议（Protocol）用于 Kalico 主机软件和 Kalico 微控制器（MCU - Micro-Controller Unit）软件之间的低级通信。高层来说，协议可以被认为是一系列被压缩、传输然后在接收端处理的命令和响应字符串。一个未压缩的人类可读格式的命令序列示例可能看起来像这样：

    set_digital_out pin=PA3 value=1
set_digital_out pin=PA7 value=1
schedule_digital_out oid=8 clock=4000000 value=0
queue_step oid=7 interval=7458 count=10 add=331
queue_step oid=7 interval=11717 count=4 add=1281

    有关可用命令的信息，请参见 MCU 命令文档。有关如何将 G-Code 文件翻译为相应的人类可读微控制器命令的信息，请参见调试文档。

    本页提供了 Kalico 消息协议本身的高级描述。它描述了消息是如何声明的、如何编码为二进制格式（"压缩"方案）以及如何被传输的。

    协议的目标是在主机和微控制器之间启用无错误的通信通道，该通道具有低延迟、低带宽和对微控制器而言低复杂性的特点。

    微控制器接口¶

    Kalico 传输协议可以被认为是微控制器和主机之间的一种 RPC（远程过程调用）机制。微控制器软件声明主机可以调用的命令以及它可以生成的响应消息。主机使用这些信息来命令微控制器执行动作并解释结果。

    声明命令¶

    微控制器软件通过在 C 代码中使用 DECL_COMMAND() 宏来声明"命令"。例如：

    DECL_COMMAND(command_update_digital_out, "update_digital_out oid=%c value=%c");

    上面声明了一个名为"update_digital_out"的命令。这允许主机"调用"这个命令，这将导致微控制器中的 command_update_digital_out() C 函数被执行。上面还表示该命令需要两个整数参数。当执行 command_update_digital_out() C 代码时，它将被传递一个包含这两个整数的数组——第一个对应"oid"，第二个对应"value"。

    通常，参数使用 printf() 样式语法来描述（例如，"%u"）。格式直接对应于命令的人类可读视图（例如，"update_digital_out oid=7 value=1"）。在上面的示例中，"value="是参数名称，"%c"表示参数是一个整数。在内部，参数名称仅用作文档。在这个例子中，"%c"也被用作文档来指示预期的整数大小为 1 字节（声明的整数大小不影响解析或编码）。

    微控制器构建将收集所有用 DECL_COMMAND() 声明的命令、确定它们的参数，并安排它们可被调用。

    声明响应¶

    要从微控制器向主机发送信息，将生成一个"响应"。这些既被声明又使用 sendf() C 宏来传输。例如：

    sendf("status clock=%u status=%c", sched_read_time(), sched_is_shutdown());

    上面传输了一个"status"响应消息，其包含两个整数参数（"clock"和"status"）。微控制器构建自动找到所有 sendf() 调用并为它们生成编码器。sendf() 函数的第一个参数描述了响应，其格式与命令声明相同。

    主机可以为每个响应注册一个回调函数。因此，实际上，命令允许主机调用微控制器中的 C 函数，而响应允许微控制器软件调用主机中的代码。

    sendf() 宏只应从命令或任务处理程序中调用，不应从中断或计时器中调用。代码不需要为了响应收到的命令而发出 sendf()，它在调用 sendf() 的次数上没有限制，并且它可以在任何时候从任务处理程序中调用 sendf()。

    输出响应¶

    为了简化调试，还有一个 output() C 函数。例如：

    output("The value of %u is %s with size %u.", x, buf, buf_len);

    output() 函数在使用上类似于 printf()——它旨在生成和格式化用于人类消费的任意消息。

    声明枚举¶

    枚举允许主机代码对微控制器处理为整数的参数使用字符串标识符。它们在微控制器代码中声明——例如：

    DECL_ENUMERATION("spi_bus", "spi", 0);

DECL_ENUMERATION_RANGE("pin", "PC0", 16, 8);

    在第一个例子中，DECL_ENUMERATION() 宏为任何参数名称为"spi_bus"或参数名称以"_spi_bus"为后缀的命令/响应消息定义一个枚举。对于这些参数，字符串"spi"是一个有效值，它将以零的整数值进行传输。

    也可以声明一个枚举范围。在第二个例子中，"pin"参数（或任何参数名称以"_pin"为后缀的参数）将接受 PC0、PC1、PC2、...、PC7 作为有效值。这些字符串将以整数 16、17、18、...、23 进行传输。

    声明常量¶

    常量也可以被导出。例如，以下代码：

    DECL_CONSTANT("SERIAL_BAUD", 250000);

    将导出一个名为"SERIAL_BAUD"的常量，其值从微控制器为 250000 到主机。也可以声明一个字符串常量——例如：

    DECL_CONSTANT_STR("MCU", "pru");

    低级消息编码¶

    为了实现上述 RPC 机制，每个命令和响应都被编码为二进制格式进行传输。本节描述了传输系统。

    消息块¶

    从主机到微控制器以及反向发送的所有数据都包含在"消息块"中。消息块有一个两字节头和一个三字节尾。消息块的格式是：

    <1 byte length><1 byte sequence><n-byte content><2 byte crc><1 byte sync>

    长度字节包含消息块中的字节数，包括头和尾字节（因此最小消息长度为 5 字节）。最大消息块长度目前为 64 字节。序列字节包含低位中的 4 位序列号，高位始终包含 0x10（高位为将来使用保留）。内容字节包含任意数据，其格式在下一节中描述。CRC 字节包含消息块的 16 位 CCITT CRC（循环冗余校验），包括头字节但不包括尾字节。同步字节为 0x7e。

    消息块的格式受 HDLC（高级数据链路控制）消息帧的启发。与 HDLC 类似，消息块可以选择性地在块的开始处包含额外的同步字符。与 HDLC 不同，同步字符不是成帧的独占字符，可能在消息块内容中出现。

    消息块内容¶

    从主机到微控制器发送的每个消息块在其内容中包含零个或多个消息命令序列。每个命令以一个 VLQ 编码（可变长度数量）的整数命令-id 开始，然后是给定命令的零个或多个 VLQ 参数。

    例如，以下四个命令可能被放置在单个消息块中：

    update_digital_out oid=6 value=1
update_digital_out oid=5 value=0
get_config
get_clock

    并编码为以下八个 VLQ 整数：

    <id_update_digital_out><6><1><id_update_digital_out><5><0><id_get_config><id_get_clock>

    为了编码和解析消息内容，主机和微控制器都必须同意命令 id 和每个命令具有的参数数量。因此，在上面的示例中，主机和微控制器都会知道"id_update_digital_out"始终后跟两个参数，而"id_get_config"和"id_get_clock"具有零参数。主机和微控制器共享一个"数据字典"，将命令描述（例如，"update_digital_out oid=%c value=%c"）映射到它们的整数命令-id。在处理数据时，解析器将知道在给定命令 id 之后期望特定数量的 VLQ 编码参数。

    从微控制器到主机发送的消息块的消息内容遵循相同的格式。这些消息中的标识符是"响应 id"，但它们的目的相同并遵循相同的编码规则。实际上，从微控制器发送到主机的消息块在消息块内容中从不包含超过一个响应。

    可变长度数量¶

    有关 VLQ 编码整数的通用格式，请参见 Wikipedia 文章。Kalico 使用一种编码方案，支持正整数和负整数。接近零的整数使用更少的字节进行编码，正整数通常使用比负整数更少的字节进行编码。下表显示了每个整数需要多少字节来编码：

    整数 编码大小
    -32 .. 95 1
    -4096 .. 12287 2
    -524288 .. 1572863 3
    -67108864 .. 201326591 4
    -2147483648 .. 4294967295 5

    可变长度字符串¶

    作为上述编码规则的例外，如果命令或响应的参数是动态字符串，那么参数不被编码为简单的 VLQ 整数。相反，它通过传输长度作为 VLQ 编码整数，后跟内容本身来编码：

    <VLQ encoded length><n-byte contents>

    数据字典中的命令描述允许主机和微控制器知道哪些命令参数使用简单 VLQ 编码，哪些参数使用字符串编码。

    数据字典¶

    为了在微控制器和主机之间建立有意义的通信，双方必须同意一个"数据字典"。这个数据字典包含命令和响应的整数标识符以及它们的描述。

    微控制器构建使用 DECL_COMMAND() 和 sendf() 宏的内容来生成数据字典。构建自动为每个命令和响应分配唯一的标识符。这个系统允许主机和微控制器代码无缝地使用描述性人类可读名称，同时仍然使用最小带宽。

    主机在首次连接到微控制器时查询数据字典。一旦主机从微控制器下载了数据字典，它就使用该数据字典来编码所有命令并解析来自微控制器的所有响应。因此主机必须处理动态数据字典。但是，为了保持微控制器软件简单，微控制器总是使用其静态（编译的）数据字典。

    数据字典通过向微控制器发送"identify"命令来查询。微控制器将以"identify_response"消息响应每个 identify 命令。由于这两个命令在获得数据字典之前是需要的，它们的整数 id 和参数类型在微控制器和主机中都被硬编码。"identify_response"响应 id 为 0，"identify"命令 id 为 1。除了具有硬编码的 id 之外，identify 命令及其响应的声明和传输方式与其他命令和响应相同。没有其他命令或响应是硬编码的。

    传输的数据字典本身的格式是一个 zlib 压缩的 JSON 字符串。微控制器构建过程生成该字符串、压缩它，并将其存储在微控制器闪存的文本段中。数据字典可能远大于最大消息块大小——主机通过发送多个 identify 命令来下载它，请求数据字典的渐进块。一旦获得所有块，主机将汇总块、解压缩数据并解析内容。

    除了通信协议的信息外，数据字典还包含软件版本、枚举（由 DECL_ENUMERATION 定义）和常量（由 DECL_CONSTANT 定义）。

    消息流¶

    从主机发送到微控制器的消息命令旨在是无错误的。微控制器将检查每个消息块中的 CRC 和序列号，以确保命令准确且有序。微控制器始终按顺序处理消息块——如果收到一个无序块，它将丢弃它和任何其他无序块，直到收到具有正确序列的块。

    低级主机代码为发送到微控制器的丢失和损坏消息块实现了自动重新传输系统。为了便于这一点，微控制器在成功接收每个消息块后传输一个"ack 消息块"。主机在发送每个块后计划一个超时，如果在没有接收到相应"ack"的情况下超时到期，它将重新传输。此外，如果微控制器检测到损坏或无序的块，它可能会传输一个"nak 消息块"以便快速重新传输。

    "ack"是一个内容为空的消息块（即，一个 5 字节的消息块），序列号大于最后接收的主机序列号。"nak"是一个内容为空的消息块，序列号小于最后接收的主机序列号。

    该协议促进了一个"窗口"传输系统，使得主机可以在任何时候有许多正在进行中的消息块。（这是除了给定消息块中可能存在的许多命令之外。）这允许即使在存在传输延迟的情况下也能最大化带宽利用率。超时、重新传输、窗口和 ack 机制受到 TCP（传输控制协议）中类似机制的启发。

    在另一个方向上，从微控制器发送到主机的消息块被设计为无错误的，但它们不保证传输。（响应不应该损坏，但它们可能会丢失。）这样做是为了保持微控制器中的实现简单。响应没有自动重新传输系统——高级代码预计能够处理偶发的丢失响应（通常通过重新请求内容或设置经常性的响应传输计划）。从主机发送的消息块中的序列号字段总是比从主机接收的消息块的最后接收序列号大一。它不用于跟踪响应消息块的序列。
