# 为 Kalico 做出贡献

感谢您为 Kalico 做出贡献！本文档描述了为 Kalico 做出贡献的流程。

请参阅 [联系页面](Contact.md) 以了解有关报告问题或与开发人员联系的详细信息。

## 贡献流程概述

对 Kalico 的贡献通常遵循高级流程：

1. 提交者通过创建 [GitHub 拉取请求](https://github.com/KalicoCrew/kalico/pulls) 开始，当提交准备好进行广泛部署时。
2. 当 [审查者](#审查者) 可用来 [审查](#审查中应该期望的) 提交时，他们将在 GitHub 上自己分配给拉取请求。审查的目标是查找缺陷并检查提交是否遵循记录的指南。
3. 成功审查后，审查者将在 GitHub 上 "批准审查"，[维护者](#审查者) 将把更改提交到 Kalico `main` 分支。

在进行增强时，请考虑在 [Kalico Discord 服务器](Contact.md#discord) 上开始（或贡献）一个讨论话题。
论坛上的持续讨论可以提高开发工作的可见性，可能会吸引对新工作感兴趣的测试者。

## 审查中应该期望的

对 Kalico 的贡献在合并前会进行审查。审查流程的主要目标是检查缺陷并检查提交是否遵循 Kalico 文档中指定的指南。

可以理解的是，有许多方法来完成任务；审查的意图不是讨论 "最佳" 实现。在可能的情况下，侧重于事实和测量的审查讨论是可取的。

大多数提交都将导致来自审查的反馈。准备好获得反馈、提供进一步的详细信息，并在需要时更新提交。

审查者通常会查看的常见事项：

1. 提交是否不存在缺陷，是否已准备好广泛部署？

   提交者应在提交前测试其更改。审查者会查找错误，但他们通常不会测试提交。已接受的提交通常在被接受后的几周内部署到数千台打印机。因此，提交的质量被认为是优先事项。

   主 [KalicoCrew/kalico](https://github.com/KalicoCrew/kalico) GitHub 存储库可能会接受实验工作，但我们鼓励提交者应在自己的存储库中进行实验、调试和测试。[Kalico Discord 服务器](Contact.md#discord) 是提高新工作意识和寻找有兴趣提供真实反馈的用户的好地方。

   提交必须通过所有 [回归测试用例](Debugging.md)。

   修复代码中的缺陷时，提交者应对该缺陷的根本原因有大致了解，修复应针对该根本原因。

   代码提交不应包含过多的调试代码、调试选项或运行时调试日志。

   代码中的注释应侧重于增强代码维护。提交不应包含 "注释掉的代码" 或过多的注释描述过去的实现。不应有过多的 "待办" 注释。

   文档更新不应声明它们是 "进行中的工作"。

2. 提交是否为在执行真实任务的真实用户提供 "高影响" 的好处？

   审查者需要至少在自己的脑海中确定，大约 "目标受众是谁"、大约 "该受众的规模"、他们将获得的 "好处"、"好处如何衡量" 和 "这些测量测试的结果"。在大多数情况下，这对提交者和审查者都是显而易见的，在审查期间不会明确陈述。

   对主 Kalico 分支的提交应该对真实受众有显著的价值。作为一般的 "经验法则"，提交应该针对至少 100 个真实用户的用户基数。

   如果审查者要求提交的 "好处" 的详细信息，请不要认为这是批评。能够理解变化的真实好处是审查的自然部分。

   在讨论优势时，最好讨论 "事实和测量"。总的来说，审查者不在寻找形式为 "某人可能会发现选项 X 有用" 的回应，也不在寻找形式为 "此提交添加了固件 X 实现的功能" 的回应。相反，通常最好讨论质量改进如何被测量的详细信息和这些测量的结果 - 例如，"Acme X1000 打印机上的测试显示改进的角，如图片所示..."，或例如 "真实对象 X 在 Foomatic X900 打印机上的打印时间从 4 小时减少到 3.5 小时"。可以理解，这类型的测试可能需要大量时间和精力。Kalico 的一些最著名的功能在被合并到 `main` 分支之前经历了数月的讨论、返工、测试和文档编制。

   所有新模块、配置选项、命令、命令参数和文档都应具有 "高影响"。我们不想用他们无法合理配置的选项来负担用户，也不想用无法提供显著好处的选项来负担他们。

   审查者可能会要求澄清用户如何配置选项 - 理想的回应将包含有关流程的详细信息 - 例如，"MegaX500 的用户预期设置选项 X 为 99.3，而 Elite100Y 的用户应该使用流程 ... 校准选项 X"。

   如果选项的目标是使代码更模块化，那么优先使用代码常数而不是面向用户的配置选项。

   新模块、新选项和新参数不应提供与现有模块类似的功能 - 如果差异是任意的，那么最好利用现有系统或重构现有代码。

3. 提交的版权是否清楚、非武断的且兼容的？

   新的 C 文件和 Python 文件应具有明确的版权声明。查看现有文件了解首选格式。在对该文件进行次要更改时，不鼓励在现有文件上声明版权。

   来自第三方来源的代码必须与 Kalico 许可证（GNU GPLv3）兼容。大型第三方代码添加应添加到 `lib/` 目录（并遵循 [lib/README](../lib/README) 中描述的格式）。

   提交者必须提供使用其完整真实姓名的 [签署行](#提交消息格式)。它表明提交者同意 [开发者原产地证书](developer-certificate-of-origin)。

4. 提交是否遵循 Kalico 文档中指定的指南？

   特别是，代码应遵循 [Code_Overview.md](Code_Overview.md) 中的指南，配置文件应遵循 [Example_Configs.md](Example_Configs.md) 中的指南。

5. Kalico 文档是否已更新以反映新更改？

   最少，参考文档必须随代码更改进行更新：
   * 所有命令和命令参数都必须在 [G-Codes.md](G-Codes.md) 中记录。
   * 所有面向用户的模块及其配置参数都必须在 [Config_Reference.md](Config_Reference.md) 中记录。
   * 所有导出的 "状态变量" 都必须在 [Status_Reference.md](Status_Reference.md) 中记录。
   * 所有新 "webhook" 及其参数都必须在 [API_Server.md](API_Server.md) 中记录。
   * 任何做出非向后兼容更改的命令或配置文件设置的变更都必须在 [Config_Changes.md](Config_Changes.md) 中记录。

   新文档应添加到 [Overview.md](Overview.md)，并添加到网站索引 [docs/_kalico/mkdocs.yml](../docs/_kalico/mkdocs.yml)。

6. 提交是否格式良好、解决单个主题，并且是独立的？

   提交消息应遵循 [首选格式](#提交消息格式)。

   提交不得具有合并冲突。对 Kalico `main` 分支的新增总是通过 "rebase" 或 "squash and rebase" 完成。提交者通常不需要在每次对 Kalico `main` 分支的更新时重新合并其提交。但是，如果存在合并冲突，则建议提交者使用 `git rebase` 来解决冲突。

   每个提交应解决单个高级变更。大型变更应分解为多个独立的提交。每个提交应该 "独立存在"，以便像 `git bisect` 和 `git revert` 这样的工具可以可靠地工作。

   空格更改不应与功能更改混合。一般来说，除非来自已建立的代码 "所有者"，否则不接受任意空格更改。

Kalico 在 Python 代码上实现了软严格的 "编码样式指南"。对现有代码的修改应遵循高级代码流、代码缩进样式和该现有代码的格式。

审查的目标不是讨论 "更好的实现"。但是，如果审查者很难理解提交的实现，那么他们可能会要求更改以使实现更加透明。特别是，如果审查者无法确信提交不存在缺陷，那么可能需要更改。

作为审查的一部分，审查者可能会为主题创建替代拉取请求。这可能是为了避免在次要程序项上过度 "来回"，从而简化提交流程。这也可能是因为讨论激励了审查者构建替代实现。这两种情况都是审查的正常结果，不应被视为对原始提交的批评。

### 帮助审查

我们感谢审查方面的帮助！不需要成为 [列出的审查者](#审查者) 才能进行审查。还鼓励 GitHub 拉取请求的提交者审查自己的提交。

要帮助进行审查，请按照 [审查中应该期望的](#审查中应该期望的) 中概述的步骤来验证提交。审查完成后，在 GitHub 拉取请求中添加注释，说明您的发现。如果提交通过了审查，请明确声明此事 - 例如 "我根据 CONTRIBUTING 文档中的步骤审查了此更改，对我来说一切看起来都很好"。如果无法完成审查中的某些步骤，请明确说明审查了哪些步骤以及哪些步骤未审查 - 例如 "我没有检查代码是否存在缺陷，但我审查了 CONTRIBUTING 文档中的所有其他内容，看起来很好"。

我们也感谢提交的测试。如果代码已测试，请在 GitHub 拉取请求中添加评论，说明您的测试结果 - 成功或失败。请明确说明代码已测试并且结果 - 例如 "我在我的 Acme900Z 打印机上使用花瓶打印测试了此代码，结果很好"。

## 提交消息格式

每个提交应具有类似于以下内容的格式化的提交消息：

```
模块：大写的、简短的（50 个字符或更少）摘要

更详细的解释文本（如果需要）。将其换行到大约 75 个字符左右。在某些情况下，第一行被视为电子邮件的主题，其余文本作为正文。摘要和正文之间的空行至关重要（除非您完全省略正文）；如果您将两者放在一起，rebase 之类的工具可能会感到困惑。

进一步的段落在空行之后..

签署者：我的名字 <myemail@example.org>
```

在上面的示例中，`模块` 应该是存储库中文件或目录的名称（不带文件扩展名）。例如，`clocksync: 在连接时修复 pause() 调用中的打字错误`。在提交消息中指定模块名称的目的是帮助为提交注释提供上下文。

在每个提交上有 "签署者" 行很重要 - 它证明您同意 [开发者原产地证书](developer-certificate-of-origin)。它必须包含您的真实姓名（抱歉，没有假名或匿名贡献）和包含当前电子邮件地址。
